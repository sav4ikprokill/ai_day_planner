# Оценка качества парсинга AI Day Planner

## Цель

Этот документ нужен для предзащиты как минимальная доказательная база того, что сценарий создания задач естественным языком оценивался не только “на глаз”, но и через небольшой ручной evaluation-набор.

Это не production-grade NLP benchmark и не научная публикация. Это честный MVP-подход: фиксированный набор из 30 пользовательских фраз, ручной прогон через `POST /tasks/parse`, заполнение фактических результатов и расчёт простых метрик качества.

## Evaluation set

Набор покрывает:
- простые задачи без времени;
- задачи с датой;
- задачи со временем;
- формулировки “сегодня / завтра / послезавтра”;
- категории `спорт`, `учёба`, `работа`, `покупки`, `здоровье`;
- явный и неявный приоритет;
- неоднозначные фразы;
- голосовые формулировки в стиле “добавь...”;
- несколько английских фраз.

## Metrics

Ниже указаны простые метрики, которые можно посчитать после ручного прогона.

### 1. Title accuracy

`title accuracy = количество фраз, где название задачи извлечено корректно / общее количество фраз`

Пример смысла:
- если parser правильно выделил основную задачу `тренировка`, `купить продукты`, `подготовить отчёт`, это считается успехом;
- если title получился пустым, бессмысленным или потерял основное действие, это ошибка.

### 2. Date/time accuracy

`date/time accuracy = количество фраз, где дата и/или время извлечены корректно / количество фраз, где дата и/или время вообще ожидались`

Важно:
- для фраз без даты и времени эта метрика не считается;
- если ожидалось `завтра 19:00`, а parser вернул только `завтра`, это можно отметить как частичную ошибку в `Notes`.

### 3. Category accuracy

`category accuracy = количество фраз, где категория определена корректно / количество фраз, где категория ожидалась`

Категории в этом наборе:
- спорт
- учёба
- работа
- покупки
- здоровье

Если parser не поддерживает явную категоризацию для некоторых фраз, это нужно честно фиксировать в `Notes`.

### 4. Priority accuracy

`priority accuracy = количество фраз, где приоритет определён корректно / количество фраз, где приоритет ожидался`

Приоритет особенно полезно проверять на фразах:
- `срочно`
- `высокая важность`
- `низкий приоритет`
- `средний приоритет`
- `важная ...`

### 5. Exact match

`exact match = количество фраз, где корректны сразу title, date/time, category и priority / общее количество фраз`

Это самая строгая метрика. Она показывает, сколько фраз parser разобрал полностью без заметных потерь.

### 6. Fallback success rate

`fallback success rate = количество фраз, которые успешно обработались без падения сервиса и без пустого результата при отключённом Gemini / количество фраз, прогнанных в fallback-режиме`

Эта метрика особенно полезна для честного ответа комиссии:
- даже если качество fallback ниже, важно показать, что core flow не ломается.

## Как выставлять Score

Можно использовать простую шкалу:
- `1.0` — всё ожидаемое извлечено корректно;
- `0.5` — результат частично корректен, но есть заметная ошибка;
- `0.0` — результат некорректен или полезного разбора не произошло.

Если хочется упростить оценку, можно использовать только:
- `pass`
- `partial`
- `fail`

## What to pay attention to during evaluation

После ручного прогона полезно отдельно выписать типичные ошибки:
- теряется глагол действия в title;
- не распознаются относительные даты;
- время извлекается без даты или наоборот;
- категория остаётся `general`, хотя пользовательская фраза явно предметная;
- приоритет игнорируется;
- voice-style фразы с `добавь ...` парсятся хуже обычных;
- английские фразы не поддерживаются или поддерживаются частично.

## How to run evaluation manually

1. Запустить core stack:

```bash
docker compose up --build -d postgres api worker web-platform
```

2. Открыть Swagger:

```text
http://127.0.0.1:8000/docs
```

3. Для каждой фразы из таблицы вызвать `POST /tasks/parse`.

4. Заполнить поля:
- `Actual title`
- `Actual date/time`
- `Actual category`
- `Actual priority`
- `Score`
- `Notes`

5. Посчитать метрики:
- title accuracy
- date/time accuracy
- category accuracy
- priority accuracy
- exact match
- fallback success rate

6. Отдельно зафиксировать типичные ошибки и слабые места.

7. При необходимости повторить тот же набор:
- один раз с `GEMINI_API_KEY`;
- один раз без `GEMINI_API_KEY`, чтобы оценить fallback.

## What to say to the committee

Короткий честный ответ:

> Для MVP я использовал небольшой ручной evaluation-набор из 30 фраз.  
> Он покрывает простые задачи, относительные даты, время, категории, приоритет, голосовые формулировки и несколько английских примеров.  
> Я прогоняю эти фразы через `POST /tasks/parse`, заполняю фактические результаты и считаю простые метрики: точность title, даты/времени, категории, приоритета, exact match и устойчивость fallback-режима.  
> Это не production-grade NLP benchmark, а честная ручная оценка качества для дипломного MVP.

## Чего не нужно заявлять

На предзащите не стоит говорить:
- что parser имеет “высокую точность”, если это не измерено;
- что качество NLP сопоставимо с production-системами;
- что английский язык полностью поддерживается, если это не подтверждено прогоном;
- что fallback parser “умный”, если по факту это упрощённый безопасный режим.

Лучше говорить так:
- качество парсинга оценивалось на небольшом ручном наборе;
- результаты фиксировались вручную;
- слабые места известны;
- для MVP важнее было сохранить core flow и деградацию без падения системы.
