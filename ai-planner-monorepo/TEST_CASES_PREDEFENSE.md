# Предзащитные test cases AI Day Planner

## Цель

Эти тест-кейсы нужны, чтобы показать комиссии, что AI Day Planner проверялся как система, а не только как набор экранов.

Они покрывают:
- запуск core stack;
- создание задачи текстом;
- создание задачи голосом;
- появление задачи в списке;
- смену статуса;
- создание привычки;
- fallback без Gemini;
- базовую доступность API.

## Scope

### Core

- FastAPI backend
- PostgreSQL
- React web demo
- Go worker
- привычки
- голосовой ввод в web demo

### Out of scope for main demo

- Java reports worker
- C++ scheduler
- production-grade load testing

## TC-01. Запуск core stack

**Цель:** проверить, что core MVP поднимается локально.

**Шаги:**
1. Выполнить `docker compose up --build -d postgres api worker web-platform`
2. Выполнить `docker compose ps`
3. Открыть `http://127.0.0.1:5173`
4. Открыть `http://127.0.0.1:8000/docs`

**Ожидаемый результат:**
- frontend доступен;
- Swagger доступен;
- core-сервисы запущены.

## TC-02. Создание задачи текстом

**Цель:** проверить главный пользовательский сценарий.

**Шаги:**
1. Открыть web demo
2. Ввести `купить продукты завтра в 18:00`
3. Нажать создание задачи
4. Найти задачу в списке

**Ожидаемый результат:**
- задача создаётся;
- задача отображается в списке;
- текстовый сценарий понятен и воспроизводим.

## TC-03. Создание задачи голосом

**Цель:** проверить browser-side voice input.

**Шаги:**
1. Открыть web demo в Chrome или Edge
2. Нажать `Голосовой ввод`
3. Произнести `добавь тренировку завтра в 19:00`
4. Проверить, что распознанный текст подставился в поле
5. Вручную нажать создание задачи

**Ожидаемый результат:**
- браузер распознаёт речь;
- текст появляется в поле;
- автосабмита нет;
- задача создаётся только после ручного подтверждения.

## TC-04. Ограничение голосового ввода

**Цель:** проверить честную обработку unsupported/error сценариев.

**Шаги:**
1. Открыть приложение в неподдерживаемом браузере или запретить микрофон
2. Нажать `Голосовой ввод`

**Ожидаемый результат:**
- пользователь видит понятное сообщение;
- система предлагает использовать текстовый ввод;
- основная функция не ломается.

## TC-05. Изменение статуса задачи

**Цель:** проверить, что пользователь может управлять задачей после создания.

**Шаги:**
1. Создать задачу
2. Найти её в списке
3. Изменить статус

**Ожидаемый результат:**
- статус меняется;
- обновление видно в UI.

## TC-06. Создание привычки

**Цель:** проверить второй важный пользовательский сценарий.

**Шаги:**
1. Перейти в раздел привычек
2. Создать привычку `спорт вечером`
3. Проверить, что привычка отображается в интерфейсе

**Ожидаемый результат:**
- привычка создаётся;
- пользователь видит результат.

## TC-07. Fallback без Gemini

**Цель:** проверить устойчивость core flow без внешнего AI.

**Шаги:**
1. Запустить backend без `GEMINI_API_KEY`
2. Создать задачу естественным языком

**Ожидаемый результат:**
- backend не падает;
- создаётся задача через fallback parser;
- основной demo-сценарий сохраняется.

## TC-08. Базовая API доступность

**Цель:** проверить, что API доступен для демонстрации.

**Шаги:**
1. Открыть `/docs`
2. Открыть `/health`
3. При необходимости вызвать `POST /tasks/parse`

**Ожидаемый результат:**
- `/docs` доступен;
- `/health` возвращает `ok`;
- parser endpoint отвечает.

## Smoke checklist перед показом

- `docker compose config`
- `docker compose up --build -d postgres api worker web-platform`
- `pnpm build`
- `pnpm typecheck`
- текстовое создание задачи
- голосовой ввод в Chrome/Edge
- смена статуса
- создание привычки
- Swagger `/docs`

## Что говорить комиссии

- Основной сценарий проверялся не только визуально, но и как последовательность конкретных шагов.
- Голосовой ввод тестируется отдельно от текстового сценария, потому что он зависит от Web Speech API.
- При недоступности внешнего AI система должна сохранять core flow через fallback parser.
