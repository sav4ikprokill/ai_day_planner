# Предзащитные тест-кейсы AI Day Planner

## 1. Цель проверок

Эти тест-кейсы нужны, чтобы показать комиссии, что AI Day Planner проверяется как система, а не как набор несвязанных экранов. Они покрывают запуск core-стека, создание задачи естественным языком, AI/fallback parsing, запись в PostgreSQL, работу jobs queue и поведение при недоступности внешних сервисов.

Для микросервисного проекта недостаточно просто открыть frontend. Даже если UI загрузился, это ещё не доказывает, что backend принимает запросы, данные сохраняются в базе, создаются фоновые jobs и worker корректно обрабатывает demo-path.

## 2. Область проверки

### Core MVP

- FastAPI backend
- PostgreSQL
- React frontend
- Telegram Mini App / PWA, если используется в demo
- Go worker
- Gemini AI parser
- local fallback parser
- PostgreSQL jobs queue

### Experimental

- Java reports worker
- C++ scheduler

Experimental-сервисы не являются обязательным путём предзащитного demo. Их можно упоминать как расширения архитектуры.

## 3. Тест-кейсы

### TC-01. Проверка core-запуска

**Цель:**  
Проверить, что core MVP поднимается в воспроизводимом виде через Docker Compose.

**Предусловия:**  
Docker установлен и доступен. Пользователь находится в корне `ai-planner-monorepo`.

**Шаги:**  
1. Выполнить `docker compose down -v`.
2. Выполнить `docker compose up --build -d postgres api worker web-platform`.
3. Выполнить `docker compose ps`.
4. Открыть `http://127.0.0.1:8000/docs`.
5. Открыть `http://127.0.0.1:5173`.

**Ожидаемый результат:**  
API открывается, frontend открывается, основные core-сервисы запущены и доступны.

**Что говорить комиссии:**  
Этот тест подтверждает, что диплом можно показать как целостную систему: есть backend, база данных, worker и пользовательский интерфейс, а не только локально запущенный frontend.

### TC-02. Создание задачи естественным языком

**Цель:**  
Проверить основной пользовательский сценарий проекта.

**Предусловия:**  
Core stack уже запущен. Для demo доступен frontend или API.

**Шаги:**  
1. Открыть frontend или отправить запрос в `POST /tasks/parse`.
2. Использовать тестовые данные: `тренировка завтра в 19:00`.
3. Проверить ответ backend.
4. Проверить список задач во frontend.
5. Проверить запись в PostgreSQL.

**Ожидаемый результат:**  
Backend принимает текст, Gemini или fallback parser возвращает структурированную задачу, задача появляется во frontend и сохраняется в PostgreSQL.

**Что говорить комиссии:**  
Это центральный demo-path проекта: пользователь не заполняет форму вручную, а описывает задачу естественным языком, после чего система переводит её в структуру, пригодную для хранения и последующей обработки.

### TC-03. Проверка fallback parser без Gemini

**Цель:**  
Показать устойчивость core flow при отсутствии внешнего AI API.

**Предусловия:**  
`GEMINI_API_KEY` не задан.

**Шаги:**  
1. Запустить backend без `GEMINI_API_KEY`.
2. Отправить текстовую задачу через `POST /tasks/parse`.
3. Проверить, создалась ли задача.
4. Проверить, что backend не завершился с ошибкой.

**Ожидаемый результат:**  
Backend не падает, используется local fallback parser, задача создаётся, качество разбора может быть ниже, чем при Gemini.

**Что говорить комиссии:**  
В дипломном MVP важно не только качество AI-разбора, но и устойчивость системы. Если внешний AI недоступен, core-сценарий создания задачи всё равно сохраняется.

### TC-04. Проверка PostgreSQL jobs queue и Go worker

**Цель:**  
Проверить асинхронную часть системы после создания задачи.

**Предусловия:**  
Core stack запущен, задача уже создана.

**Шаги:**  
1. Создать новую задачу.
2. Проверить таблицу `jobs` в PostgreSQL.
3. Проверить логи Go worker.
4. Проверить итоговый статус job.

**Ожидаемый результат:**  
После создания задачи появляется запись в `jobs`, worker забирает job, job получает статус `completed`, а при отсутствии SMTP/Telegram core flow не ломается.

**Что говорить комиссии:**  
Этот тест показывает, что проект не заканчивается на HTTP-запросе. В системе есть отдельный асинхронный слой, где фоновые действия отделены от пользовательского API.

### TC-05. Проверка устойчивости к отсутствию внешних сервисов

**Цель:**  
Проверить поведение системы в деградационных режимах.

**Предусловия:**  
Core stack доступен.

**Шаги:**  
1. Проверить сценарий без Gemini API.
2. Проверить сценарий без Telegram bot token.
3. Проверить сценарий без настроенного SMTP.
4. Проверить сценарий без поднятого C++ scheduler.

**Ожидаемый результат:**  
Без Gemini используется fallback parser. Без Telegram можно показывать web/PWA. Без SMTP можно показывать jobs и логи worker. Без C++ scheduler используется fallback sorting или `optimized` endpoint остаётся демонстрационным.

**Что говорить комиссии:**  
Для предзащиты это особенно важно: проект должен переживать отсутствие части внешней инфраструктуры и всё равно сохранять основной путь демонстрации.

### TC-06. Оценка нагрузки и лимитов Gemini

**Цель:**  
Честно описать эксплуатационные ограничения AI API.

**Предусловия:**  
Есть понимание, что текущий проект — дипломный MVP, а не production-нагрузочная система.

**Шаги:**  
1. Проверить актуальные лимиты Gemini в Google AI Studio.
2. Уточнить ограничения по requests per minute.
3. Уточнить ограничения по tokens per minute.
4. Уточнить ограничения по requests per day.
5. Оценить примерную нагрузку как дневной лимит, делённый на среднее число AI-задач на одного пользователя.
6. Проверить, что при превышении лимитов система может использовать fallback parser.

**Ожидаемый результат:**  
Понятно, что Gemini API имеет лимиты `requests per minute`, `tokens per minute`, `requests per day`, бесплатные и текущие лимиты могут меняться, их нужно проверять перед реальным запуском, а при превышении лимитов должен использоваться fallback parser.

**Что говорить комиссии:**  
Текущая версия проекта рассчитана на демонстрационную и малую пользовательскую нагрузку. Для production-нагрузки нужны отдельные load tests, rate limiting, monitoring и, возможно, платный тариф AI API.

**Предусловия:**

**Шаги:**  
1. Объяснить, что проект не пытается заменить массовые коммерческие продукты.
2. Показать, что основная цель диплома — реализовать собственную интеллектуальную систему планирования.
3. Выделить отличия от обычного todo-list.

**Ожидаемый результат:**  
Позиционирование проекта остаётся честным: это не “убийца Todoist”, а дипломный MVP интеллектуального планировщика.

**Что говорить комиссии:**  
Проект не пытается заменить Google Calendar, Todoist, TickTick или Notion. Его цель — показать собственную интеллектуальную систему планирования. Отличие от обычного todo-list: ввод естественным языком, AI/fallback parsing, async jobs, worker, Telegram Mini App/PWA и расширяемая архитектура.

### TC-08. Доверие к AI

**Цель:**  
Объяснить границы использования AI в проекте.

**Предусловия:**  
Комиссия задаёт вопрос о надёжности AI и риске ошибок.

**Шаги:**  
1. Показать, что AI используется на этапе ввода задачи.
2. Показать, что пользователь видит результат.
3. Показать, что пользователь может изменить или удалить задачу.
4. Объяснить роль fallback parser.

**Ожидаемый результат:**  
Понятно, что AI не принимает критические решения за пользователя, а используется как помощник ввода. Пользователь может проверить, изменить или удалить задачу, а fallback parser сохраняет базовый режим без AI.

**Что говорить комиссии:**  
В проекте AI не заменяет пользователя, а снижает ручной ввод.

### TC-09. Работа в России и региональные ограничения

**Цель:**  
Показать понимание рисков, связанных с доступностью внешних AI и платформенных сервисов.

**Предусловия:**  
Комиссия задаёт вопрос о работе в России или в условиях сетевых ограничений.

**Шаги:**  
1. Объяснить, что доступность Gemini API зависит от региона, сети, аккаунта и политики провайдера.
2. Показать, что при недоступности Gemini система использует local fallback parser.
3. Объяснить, что Telegram/PWA сценарий зависит от доступности Telegram и хостинга.
4. Описать возможное будущее развитие через интерфейс AI-provider.

**Ожидаемый результат:**

**Что говорить комиссии:**

## 5. Короткие ответы на вопросы комиссии

### Почему PostgreSQL queue, а не RabbitMQ/Kafka?

Для дипломного MVP PostgreSQL queue проще, транзакционна и не требует лишней инфраструктуры. `FOR UPDATE SKIP LOCKED` позволяет безопасно разбирать jobs несколькими worker’ами. RabbitMQ/Kafka оправданы при большей нагрузке и отдельной event-driven архитектуре.

### Сколько пользователей выдержит система?

Без load tests нельзя честно назвать точное число. Текущий проект рассчитан на demo/small load. Ограничения: Gemini API quota, PostgreSQL, backend resources и worker throughput.

### Что будет, если Gemini закончится?

Сработает fallback parser. Качество разбора будет ниже, но создание задачи сохранится.

### Почему можно доверять AI?

AI не принимает окончательные решения, а только помогает преобразовать текст в структуру. Пользователь видит результат и может его изменить.

### Как проект будет работать в России?

Core-сценарий не должен зависеть только от Gemini: при недоступности AI используется local fallback parser. Для production в конкретном регионе нужно выбирать доступного AI-провайдера и хостинг.

### Зачем проект, если есть Todoist/Google Calendar?

Цель диплома — не победить коммерческие продукты, а реализовать и показать собственную архитектуру интеллектуального планировщика: AI parsing, async jobs, worker, PWA/Telegram и микросервисное разделение.

## 6. Финальная формулировка для предзащиты

AI Day Planner — это дипломный MVP микросервисной системы персонального планирования дня. Система позволяет создавать задачи естественным языком, преобразует текст в структурированную задачу через Gemini или local fallback parser, сохраняет данные в PostgreSQL и обрабатывает фоновые действия через PostgreSQL jobs queue и Go worker. Проект демонстрирует AI-интеграцию, устойчивость к внешним ограничениям, async processing и несколько клиентских интерфейсов.
